Tool Timeout Budgets for AI Coding Agents Without Hung Runs

June 23, 2026 • 10 min read • Agent Reliability

A lot of agent failures are not really reasoning failures. They are timeout failures with good marketing.

A shell command waits forever on a package mirror. A browser step hangs behind a modal. A model gateway streams one token and then stalls. The agent looks indecisive, but the real bug is that nobody taught the tool lane when to give up.

This post shows how I would design timeout budgets for coding agents so slow tools degrade predictably instead of consuming the entire run. The pattern is layered deadlines, explicit cancellation, retry classes, and fallback lanes that match the value of the step.

Why this matters

If your agent has a 20 minute task budget and one verifier command can sit for 18 minutes before dying, you do not have a verifier. You have a single-point schedule risk.

Timeout policy gets more important as agents get more autonomous. Parallel runs amplify queue contention, browser steps hide stalls better than CLI tools, and API retries can quietly turn a 10 second step into a 4 minute one.

Best practice: The practical goal is not to make every tool fast. It is to make every delay legible.

Architecture or workflow overview

  1. Task budget for the whole run.
  2. Step budget for the current unit of work.
  3. Tool budget for the exact command, API call, or browser action.
  4. Fallback policy when the tool exceeds budget.

Implementation details

1. Put deadlines in the tool manifest

# tool-budgets.yaml
defaults:
  soft_timeout_ms: 15000
  grace_timeout_ms: 5000
  hard_timeout_ms: 25000
  max_retries: 1

classes:
  quick_read:
    soft_timeout_ms: 4000
    grace_timeout_ms: 1000
    hard_timeout_ms: 6000
    max_retries: 0

  verify_medium:
    soft_timeout_ms: 30000
    grace_timeout_ms: 10000
    hard_timeout_ms: 45000
    max_retries: 1

tools:
  npm_test_changed:
    class: verify_medium
  git_diff_name_only:
    class: quick_read

This keeps timeouts out of scattered shell one-liners and makes it obvious when a tool is using the wrong class.

2. Resolve per-step deadlines from remaining task time

function resolveBudget(tool, remainingTaskMs, remainingStepMs) {
  const ceiling = Math.max(1500, Math.min(remainingTaskMs, remainingStepMs));
  const hardMs = Math.min(tool.hardMs, ceiling);
  const graceMs = Math.min(tool.graceMs, Math.floor(hardMs * 0.35));
  const softMs = Math.max(500, Math.min(tool.softMs, hardMs - graceMs));

  return {
    softMs,
    graceMs,
    hardMs,
    maxRetries: hardMs < tool.hardMs ? 0 : tool.maxRetries,
  };
}

A 45 second verifier can be reasonable at the start of a run and reckless when only 18 seconds remain.

3. Soft timeout, grace timeout, then hard kill

async function runWithBudget(cmd, budget) {
  const child = spawn(cmd[0], cmd.slice(1), { stdio: ['ignore', 'pipe', 'pipe'] });

  const softTimer = setTimeout(() => child.kill('SIGTERM'), budget.softMs);
  const hardTimer = setTimeout(() => child.kill('SIGKILL'), budget.softMs + budget.graceMs);

  const result = await collectChildResult(child);
  clearTimeout(softTimer);
  clearTimeout(hardTimer);
  return result;
}

Soft timeout gives cooperative tools a chance to flush logs and cleanup state. Hard kill protects the run when they do not.

4. Keep a receipt the agent can cite

{
  "tool": "playwright_checkout_flow",
  "budget": { "softMs": 12000, "graceMs": 5000, "hardMs": 17000 },
  "attempt": 1,
  "stdoutHeartbeatMs": 840,
  "result": "timed_out",
  "fallback": "switch_to_api_health_check",
  "capturedTail": "waiting for selector [data-testid=checkout-submit]"
}

That receipt turns a vague failure into a reviewable decision.

$ agent verify --target checkout-flow
[budget] task=19.8m step=2.0m tool=17.0s (soft=12.0s grace=5.0s)
[tool] playwright_checkout_flow attempt=1
[warn] soft timeout reached, sending SIGTERM
[warn] grace expired, forcing SIGKILL
[fallback] switching to api_health_check + screenshot artifact review
[summary] verifier degraded safely, patch review may continue with medium confidence

What went wrong and the tradeoffs

The most common failure mode is retrying the wrong thing. A slow registry or flaky model gateway may justify a retry. A deadlocked migration script usually does not.

SituationGood defaultWhy
git diff, metadata readsNo retryCheap to rerun at a higher layer
HTTP 429 or 503Bounded retry with jitterProvider may recover quickly
Browser selector timeoutOne retry after page resetSession may be poisoned
Long verifier with zero stdoutNo retry, downgrade laneRepeated silence is signal
Schema migration lock waitAbort and escalateUnsafe side effects are possible
Pitfall: A timeout without cleanup can be worse than no timeout. You stop waiting, but the machine keeps doing the work anyway.

Cancellation is part of correctness. Browser workers, Docker builds, and subprocess trees often ignore polite shutdown unless you manage process groups carefully.

Security matters here too. Secrets brokers should fail closed on timeout, approval checks should never auto-pass because the policy service was slow, and destructive tools should not resume from partial state without a receipt.

Practical checklist

  • Define timeout classes for reads, writes, browsers, and verifiers.
  • Derive tool budgets from remaining task time, not static config alone.
  • Separate soft deadline, grace period, and hard kill.
  • Record retry reason and fallback lane in a machine-readable receipt.
  • Track stdout heartbeat gaps so slow and stuck are not treated the same.
  • Fail closed for approvals, secrets, and destructive operations.
  • Measure tool percentiles weekly and tune classes with data.

Conclusion

Hung runs are usually orchestration debt wearing an agent costume. Once each tool has a real budget, a kill path, and a fallback lane, the agent becomes much easier to trust and much easier to debug.
